用 llama 批量预处理字幕：结构化 Prompt 提取难词、翻译与语法

天气与目标

八月末，秋老虎来了。手机跑 llama 的时候格外烫手。

现在字幕文件有了，OCR 能识别了，llama 也能跑了——但它们还各自为政。今天的目标是把它们串起来：用户导入字幕 → 后台用 llama 逐句分析 → 提取难词、翻译、语法结构 → 缓存到数据库，播放时直接用。

这个过程的精妙之处在于：预处理是离线做的。用户导入字幕时跑一次，之后看电影就不需要实时调用 llama 了——所有分析结果都躺在数据库里，毫秒级读取。

原理

为什么要预处理而不是实时分析？

一部电影大约 2000-3000 句字幕，每句 llama 推理 0.5-2 秒，整部预处理约 15-30 分钟——用户导入后去倒杯水回来就好了。

结构化输出设计

最关键的挑战是让 llama 输出可解析的 JSON，而不是自由文本。做法：

  1. 设计一个固定格式的 Prompt 模板
  2. 要求 llama “输出 ONLY valid JSON”
  3. 在 Kotlin 中解析 JSON，容错处理（llama 偶尔会输出不完整的 JSON 或带 Markdown fence）
  4. 使用 kotlinx.serialization 做类型安全的 JSON 反序列化

可能遇到的问题

问题 1：llama 输出的 JSON 格式不对

症状：JsonDecodingException: Unexpected JSON token at offset...

原因：llama 输出自由文本而非纯 JSON。

解决：

  • 降低 temperature（用 do_sample=false 或极低 temperature）
  • 在 Prompt 中强调”ONLY valid JSON, no other text”
  • 代码中过滤 Markdown fence 和尾部乱码

问题 2：预处理太慢，30 分钟还没跑完

原因：每句 2 秒 × 3000 句 = 100 分钟。

优化：

  • 减少 maxTokens：翻译不需要 256 token，128 就够了
  • 跳过短句：如 “Yeah.”、”Okay.” 等单词语句跳过
  • 用更小的量化级别（Q3 比 Q4 快 30%）
  • 支持断点续处理（已实现——getUnprocessedEntries 只拿未处理的）

问题 3：内存不足导致 OOM

原因：llama 模型占 1.5GB + Android 系统占 1GB + 其他 App = 超出 4GB。

解决：

  • 在 n_gpu_layers 调优（减少 GPU offload 层数）
  • 用 Q2 量化（600MB 而不是 1.2GB）
  • 预处理期间建议关闭其他 App

小结

预处理管线是整个项目的”大脑”——它提前把字幕分析好，让看电影时的体验从”等 5 秒”变成了”瞬间显示”。核心技巧是用结构化 Prompt + 容错 JSON 解析来驯服 llama 的输出。

八月湖水平 涵虚混太清
批量预处理 一字一句行
llama 千句过 缓存一朝成
