天气与目标

八月中旬，立秋已过，天气却像个不肯退烧的病人——CPU 温度降不下来，人也不想出门。

今天的任务看似”就一个文件”——把 llama.cpp 编译成 JNI 动态库，让 Kotlin 能直接调 llama 的 API，在手机上跑文本生成。 实际上，这一步是 C++ 和 Java/Kotlin 之间的翻译层，搭好了就是桥，搭不好就是墙。我们从编译到调试，一步步来。

原理

为什么选 llama.cpp

跑大模型推理，移动端的选择并不多：

方案 体积 推理速度 模型格式 生态
MLC-LLM 需要 TVM Unity 快（编译优化） MLC 专用格式 较小
llama.cpp 单文件 C++，无依赖 中等（CPU 为主） GGUF 通用格式 极丰富
ExecuTorch 依赖 PyTorch 生态 中等 .pte 格式 较新
Transformers (Java) 极大 慢 HuggingFace 格式 丰富但重

对”电影英语助手”这个场景——需要在后台静默分析字幕并做出实时反应——我们有三个硬约束：

  1. 不能依赖网络——翻译和总结必须在本地完成；
  2. 不能吃太多内存——手机总共 8GB RAM，系统占了 4GB，剩下的要留给 OCR + TFLite + 大模型；
  3. 模型要小但效果要够用——1.5B 参数的 Qwen2.5，GGUF Q4_K_M 量化后约 900MB，刚好装得下。

llama.cpp 是天然的答案：纯 C/C++ 实现，零外部依赖（甚至不依赖 STL 的 iostream），GGUF 格式生态丰富（HuggingFace 上成千上万个 GGUF 模型），CPU 推理足够用，代码量小到可以通读。它的核心只有几个 .cpp 和 .h 文件，没有复杂的构建系统。

JNI 桥接架构

整个调用链分三层——Kotlin 业务层、JNI 翻译层、C++ llama.cpp 核心层：

关键设计决策：

  1. Kotlin 不直接碰 C++ 对象——用 Long 传递原生指针，Kotlin 侧只存一个 Long 句柄，所有内存管理在 C++ 侧完成。
  2. 三函数接口（init/generate/free）——最小化 JNI 边界，跨边界的调用越少越好，每次 JNI 调用都有固定开销。
  3. extern “C” 避免符号粉碎——C++ 编译器会 mangling 函数名（加类型后缀），JNI 通过函数名签名找函数，必须用 C 链接方式导出。

要点解释：

  • SHARED 类型——Android 的 JNI 库必须是 .so（动态库），不是 .a（静态库）。System.loadLibrary("llama-bridge") 加载的是 libllama-bridge.so。
  • IMPORTED 类型——libllama.a 不在 CMake 的源码树里，是外部预编译产物，所以用 IMPORTED。
  • ${ANDROID_ABI}——这个变量由 Android Gradle Plugin 注入，值取决于 build.gradle.kts 中的 ndk.abiFilters。我们只选了 arm64-v8a，所以它的值是 arm64-v8a。
  • ${log-lib}——Android 系统的 liblog.so，C++ 侧可以用 __android_log_print() 打日志到 logcat。

第三步：写 JNI 桥接层 llama-bridge.cpp

app/src/main/cpp/llama-bridge.cpp——核心文件，Kotlin 和 llama.cpp 之间的翻译官：

设计要点逐行讲：

  1. LlamaContext 结构体——llama.cpp 的三个核心对象必须保持生命周期一致。model 负责权重数据，ctx 是推理运行时状态（KV cache），vocab 是词表映射。把它们捆在一个结构体里，一个 delete 就能清理干净。

  2. llama_tokenize 的两次调用模式——第一次传 nullptr + 0、取负数返回值得知所需数组大小，分配好 vector 后再调第二次填充实际数据。这是 llama.cpp API 的标准模式，类似 POSIX 的 snprintf(NULL, 0, ...)。

  3. 贪心采样——遍历 vocab 里所有 token 的 logit 值，取最大的那个。这不是最好的采样策略（temperature sampling 或 top-p 更自然），但代码最简单、结果可重现、适合调试。后续可以换。

  4. **reinterpret_cast<jlong>(lc)**——指针在 64 位 Android 上是 8 字节，jlong 也是 8 字节，刚好安全。这个技巧让 Kotlin 侧完全不必关心 C++ 内存布局，只看到一个不透明句柄。

可能遇到的问题

问题 1：java.lang.UnsatisfiedLinkError: dlopen failed: library "libllama.a" not found

原因：APK 里没有 libllama-bridge.so，或者它的 ABI 和当前设备不匹配。

排查流程：

常见原因：abiFilters 设了 arm64-v8a 但手机是 32 位的（非常老），或者 jniLibs 目录路径不对。

问题 2：undefined symbol: llama_model_load_from_file

原因：CMake 没找到 libllama.a，或者 .a 文件里没有这个符号。

排查：

如果 .a 是 x86_64 架构而不是 aarch64，也会报 undefined symbol——因为链接器的目标架构和库文件不匹配。

问题 3：JNI DETECTED ERROR IN APPLICATION: use of deleted local reference

原因：GetStringUTFChars 拿到的 const char* 在 ReleaseStringUTFChars 之后失效。如果后续还用那个指针，就访问了已释放的内存。

解决：确保 ReleaseStringUTFChars 在最后调用——在所有对 promptStr 的访问结束之后。代码中 ReleaseStringUTFChars(prompt, promptStr) 放在 llama_decode 循环之后、NewStringUTF 之前，是正确的顺序。

问题 5：模型加载后 OOM，应用被杀

原因：1.5B Q4_K_M 的 GGUF 文件约 900MB，加载到内存后模型权重约 1.2GB，加上 KV cache（2048 上下文，每层约 3MB × 28 层 ≈ 80MB），总内存接近 1.5GB。如果手机剩余内存不够，直接 OOM。

缓解措施：

  1. 用更小的模型——Qwen2.5-0.5B Q4_K_M 约 350MB；
  2. 用 IQ2_XXS 量化——进一步压缩到约 600MB；
  3. 减小上下文长度——n_ctx = 512 足够字幕翻译，KV cache 从 80MB 降到约 20MB；

小结

把 llama.cpp 做成 JNI 库，本质上是在 C++ 和 Kotlin 之间架一座窄桥——窄到只有三个函数。窄的好处是边界清晰、出问题好排查；坏处是不能直接暴露 llama.cpp 的所有高级功能（如 grammar-based sampling、batch inference）。

但对我们”电影英语助手”的场景来说，这三个函数就够了：初始化模型 → 喂 prompt 拿结果 → 释放资源。900MB 的 GGUF 模型在手机的后台默默运行，生成翻译和解释，用户甚至感觉不到它的存在。
