模型剪枝：移除冗余通道和层，让 OCR 模型变小 40%

Created2025-11-14|Updated2026-06-08|电影英语助手
|Post Views:

今日目标

十一月中旬，南方开始降温。还记得三个月前热得开空调，如今已经换上长袖。

蒸馏后的 CRNN 模型在准确率上已经比较满意了——CER 降到 9% 以下，字幕识别基本不出大错。但有个问题一直悬在心头：模型 5MB 还是太大了。 虽然比最初的 35MB 已经小了 7 倍（感谢量化），但 Android 端的 APK 体积每一 KB 都关乎用户的下载意愿。

能不能在不损（或少损）准确率的前提下，把模型再砍掉一截？答案就是结构化剪枝。

今天的目标：用 PyTorch 内置的 prune 工具，对 CRNN 的 CNN 卷积层做结构化通道剪枝，搭配迭代剪枝和微调策略，把模型体积压缩 40%，计算量同步降低。 不玩花活，只讲能落地的步骤。

一、剪枝的分类：结构化 vs 非结构化

1.1 非结构化剪枝（权重剪枝）

问题：矩阵形状没变，存的是稀疏矩阵。但 GPU 和 TFLite 对稀疏矩阵乘法支持并不完善。稀疏推理不仅不加速，反而可能因为稀疏/稠密转换开��而变慢。

1.2 结构化剪枝（通道/滤波器剪枝）

结构化剪枝后模型是稠密小矩阵——常规矩阵乘法就天然快，不需要任何特殊硬件或运行时支持。这对 TFLite 移动端推理至关重要。

1.3 为什么我们的剪枝只针对 Conv2d？

CNN 卷积层存在大量冗余——很多通道学到的是几乎相同的滤波器模式（比如不同方向的高斯边缘）。LSTM 层的权重更精贵，剪得狠了会直接损伤序列建模能力。

二、通道重要性评分：L2-norm

2.1 评分标准

我们需要一个方法来判断”哪些通道不重要”。PyTorch 提供了 prune.ln_structured，它按 Ln-norm（默认 L2-norm，即欧几里得范数）对通道评分：

为什么 L2-norm 有效？

  • L2-norm 反映了一个滤波器权重的”幅度”
  • 幅度大的滤波器对下一层输入的影响大（信号强）
  • 幅度接近 0 的滤波器输出几乎全为 0，等于”没在工作”
  • 删除它们对模型输出的扰动最小

2.2 其他评分方法对比

评分方法 含义 优点 缺点
L2-norm 权重幅度 简单，PyTorch 内置 不一定反映梯度信息
L1-norm 权重绝对值和 倾向于产生稀疏输出 与 L2 差别不大
Taylor 展开 对 loss 的影响 最准确 计算量大，需要算梯度
随机 随机选择 最快 效果最差，纯碰运气

对于 CRNN 这种小模型，L2-norm 足够了。Taylor 方法更适合大语言模型的剪枝场景。

五、prune.remove() 的重要性

5.1 剪枝的内部机制

PyTorch 的 prune.ln_structured 调用后，模块的 weight 属性会从普通 Parameter 变成 PruningContainer（一个复合体）：

前向传播时，PyTorch 会自动用 weight_orig * weight_mask 替代 weight。所以模型能正常跑——但模型文件保存的仍然是完整大小的权重。

5.2 prune.remove() 做了什么

这一步：

  1. 计算 weight = weight_orig * weight_mask（永久化掩码效果）
  2. 把结果赋值给 module.weight（变回普通 Parameter）
  3. 删除 weight_orig 和 weight_mask

不调用 remove() 的后果：

  • 保存的 .pt 文件和剪枝前一样大（因为 weight_orig 还是原来的大小）
  • 导出的 ONNX/TFLite 模型也没有变小
  • 结构化剪枝的”减小模型体积”目标就没达成

5.3 remove() 后通道真的变少了吗？

是的。prune.remove() 后通道数真的减少了（out_channels 从 64 变成 38）。这就是结构化剪枝区别于非结构化剪枝的核心。

6.2 为什么剪枝后再量化效果特别好？

剪枝去除了大量”弱信号”通道。量化（INT8）对弱信号的误差容忍度低，小信号容易被量化噪声淹没。剪枝后模型是”少而精”的通道，每个通道的信号都较强，对量化误差不敏感。

遇到的问题

问题 1：剪完后的模型无法加载

现象：torch.load() 后 model.load_state_dict() 报 shape mismatch。

原因：剪枝后的通道数变了，但保存的 state_dict 里 key 可能是 weight_orig 而不是 weight。

解决：确保在 torch.save() 之前调用了 remove_pruning_reparam()。加载时用 strict=False 跳过形状不匹配的层（但不推荐，最好是确保保存-加载流程一致）。

问题 2：微调时 CER 不降反升

现象：微调了几个 epoch，CER 比没微调时还高。

原因：学习率太高了。剪枝后的模型处于一个”次优但尚可”的营养状态，大学习率会把参数震出这个盆地。

解决：微调学习率设为原始训练学习率的 1/10 到 1/20。

问题 3：某些层剪完后 out_channels=0

破坏性 bug：如果某层 out_channels 只有 16，sparsity=0.4 剪 40% 后还剩 10，正常。但如果 sparsity 太大（比如 0.6），可能把某层剪到只剩下几个通道，甚至为 0。

问题 4：剪枝后导出 TFLite 失败

TFLite 转换器有时不认 PruningContainer 这种复合 Parameter。确保在导出前调用了 prune.remove()。

小结

结构化剪枝 + 微调是一条成熟的模型压缩路径。它的核心优势在于：

  1. 不改变模型结构——剪枝后的模型仍然是 CRNN，只是通道数少了
  2. 不依赖特殊硬件——稠密小矩阵的乘法天然快，TFLite 完全兼容
  3. 和量化互补——剪枝后的模型对量化噪声更鲁棒
  4. 可控的精度损失——通过迭代剪枝 + 微调，可以把精度损失控制在 0.5 个 CER 以内

从 35MB (FP32 raw) → 5MB (INT8 quantized) → 最终约 3MB (pruned + quantized)，CRNN 的体积压缩了近 12 倍——这就是模型压缩的魅力。
