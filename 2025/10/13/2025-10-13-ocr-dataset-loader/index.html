OCR 数据集加载：标注 JSON 变成训练用 DataLoader

天气与目标

十月中旬，天气转凉。标注工具导出的 annotations.json 躺在那儿一周了——是时候让它上训练场了。

今天的目标：用 PyTorch 的 Dataset + DataLoader 把标注 JSON 变成模型能吃的训练数据。三步走：数据集类定义、图像增强、批处理 padding。写完这篇，前面三周标注的数据就能喂进 CRNN 了。

（代码文件路径：project/training/data/dataset.py）

原理

数据链路总览

为什么 OCR 的 DataLoader 比分类的复杂？

普通图像分类：每张图固定大小 → 直接 stack 即可。
OCR：每张图宽度不同（取决于文本长度）→ 需要 padding 到批次最大宽度后再 stack。同时文本序列长度也不同 → 文本也要 padding。

两个”不定长”叠加，让 collate_fn 成了关键。

实现步骤

第一步：字符集定义

关键设计：BLANK_IDX 是 CTC Loss 的核心概念。CTC（Connectionist Temporal Classification）需要 model 输出在字符之间插入空白符来表示”重复字符的边界”——比如 “hello” 如果不加空白符，CRNN 会把它读成 “helo”（因为连续两个 l 无法区分）。加了 BLANK 后，输出序列变成 h-e-l-l--l-o，解码时去掉重复和空白，得到正确的 “hello”。

第二步：OcrDataset 类

第三步：CTC 解码器（推理时用）

第四步：collate_fn – 批次 padding

这是整个数据管线中最精巧的部分。图片宽度不定、文本长度不定——需要在一个 batch 内统一。

第五步：组装 DataLoader + 训练/验证集划分

可能遇到的问题

问题 1：字符集之外的字被静默丢弃

[CHAR_TO_IDX[c] for c in sample["text"] if c in CHAR_TO_IDX] 这行会丢弃不在字符集中的字符。如果字幕里出现了大写字母或中文，会被直接跳过。

解决：标注时确保文本统一小写（在 AnnotatorView 保存时 toLowerCase），或在 __getitem__ 中做 .lower() 处理。当前代码已做了 .lower()。

问题 2：bbox 裁剪坐标越界

如果标注时的 bbox 坐标因为浮点精度导致略小于 0 或略大于 1，int(l * w) 可能产生负数或超出图片尺寸的坐标，导致 PIL 的 crop() 报错。

解决：代码中已用 max(0, ...) 和 min(w, ...) 做了 clamp。

问题 3：DataLoader 的 num_workers 和 Windows 兼容性

Windows 上使用 num_workers > 0 时可能出现 BrokenPipeError 或多进程启动问题。

解决：Windows 需要 if __name__ == "__main__": 包裹训练代码，或用 num_workers=0。在 Linux 上不存在此问题。可以在代码中做平台判断：

问题 4：CTC Loss 的 input_length 计算

torch.zeros(B, max_text_len) 中 padding 字符是 0（即 ‘a’ 的索引），这本身没问题，因为 CTC Loss 通过 text_lengths 知道每条文本的真实长度，不会计算 padding 部分的 loss。但如果文本为空（过滤后长度为 0），需要从数据集中移除该样本。

小结

从 annotations.json 到 DataLoader 的数据管线完成了。核心工作：Dataset 处理单样本的加载-裁剪-缩放-增强，collate_fn 处理批次的 padding。这个设计的妙处在于——图片宽度是”不定的”，但通过批次内 padding + CTC 解码，模型完全能处理任意长度的文本行。下一站：把这些数据喂进 CRNN。
