今天的目标：跑通从数据加载到模型保存的完整训练流程。 不蜻蜓点水，而是把 train.py 每一行代码掰开揉碎了讲——CTC Loss 的两个 length 参数是什么，调度器什么时候降学习率，验证集上的 CER 怎么算，early stopping 什么时候该停下来。

一、CTC Loss 核心概念

1.1 为什么普通分类损失不行？

CRNN 的输出形状是 (B, T, num_classes)，即每个样本有 T 个时间步，每个时间步输出 num_classes 个类别的 logits。假设输入一张写有 “cat” 的图片，模型输出 12 个时间步，每个时间步有 41 个类别的概率。

问题来了：我们只知道整张图的标注是 “cat”（3 个字符），但不知道每个字符对应哪几个时间步。如果强行用 CrossEntropyLoss，需要一张 对齐表，注明 t=24 是 c、t=57 是 a、t=8~10 是 t——这几乎是不可能手动标注的。

**CTC Loss 的作用就是”不需要对齐”**——你只需要告诉它最终输出 “cat”，CTC 自己会在所有可能的对齐方式中求和（或取 max），算出似然度。

1.2 input_lengths vs target_lengths

这是 CTC Loss 最容易被误解的两个参数。

  • input_lengths：每个样本经过 CNN 后产生了多少个时间步。假设输入宽度 W=200，CNN 下采样 4x，就是 50 个时间步。同一 batch 内可以不同（如果图片宽度不一致）。
  • target_lengths：每个样本的标注文本有多少个字符。”cat” 就是 3，”hello” 就是 5。绝对不能包含 zero-length target——CTC Loss 遇到长度为 0 的 target 会直接返回 NaN。

1.3 Blank Token 的角色

CTC 引入一个特殊的 “blank” token（索引 = len(CHARS)，通常排在所有真实字符后面），它的作用有两个：

  1. 充当分隔符：h-blank-h 解码为 “hh”（两个 h 被 blank 隔开，不会合并）
  2. 吸收空白区域：图片两端没有文字的区域，模型应该输出 blank

CTC 的数学核心：对所有合法对齐的路径求和（或近似），然后最大化这个概率。具体实现用的是动态规划（前向后向算法），不需要显式枚举。

1.4 zero_infinity 参数

zero_infinity=True 是训练稳定性的保险措施：当某个样本的 CTC Loss 计算结果为 inf（常见于 target_length > input_length 的情况，即字符比时间步还多），PyTorch 自动把它置为 0 而不是让整个 batch 的 loss 变成 NaN。

如果不用这个参数，一个坏样本能毁掉一个 batch 的梯度，然后整个训练过程就崩了。

三、训练过程的几个关键设计

3.1 ReduceLROnPlateau 为什么比固定衰减好？

固定衰减（如每 30 个 epoch 降为原来的 0.1）的问题在于：你不知道模型何时收敛。有时候 20 个 epoch 就停了，有时候 80 个 epoch 还在降。

ReduceLROnPlateau 是”看情况”降低学习率——只有验证指标不再改善（patience 个 epoch 内没刷新最低值）才降。这种自适应策略避免了”该降的时候没降”和”不该降的时候瞎降”。

3.2 Early Stopping 的 patience 怎么选？

early_stop_patience 不是拍脑袋决定的。我们的实验中：

  • patience=5：停得太早，模型还没收敛完
  • patience=10：正常，大部分训练在 60~80 个 epoch 停止
  • patience=20：停得太晚，后面的 epoch 基本在过拟合边缘徘徊

经验法则：先跑 100 个 epoch 不设 early stop，观察 CER 曲线的”平坦区”有多长。patience 设为平坦区的 1.5~2 倍。

3.3 模型检查点保存策略

我们只保存 Best Model，不保存每个 epoch。原因：

  1. 存储空间：一个 CRNN 模型约 35MB，100 个 epoch 就是 3.5GB
  2. 实际需要：你只需要推理性能最好的那个模型，中间的都只是过程产物
  3. torch.save 存的是 state_dict（参数字典），不是整个模型对象——更省空间，加载也更灵活

四、常见训练问题与排查

问题 1：Loss 突然变成 NaN

最常见的原因：数据集中存在零长度的标注文本（target_lengths = 0）。

CTC Loss 内部有一个概率归一化步骤——当 target_lengths 为 0 时，所有对齐路径的概率加起来是 0，归一化产生除零。虽然 zero_infinity=True 会把这个样本的 loss 置零，但如果大量样本 target_lengths=0，模型就学不到东西。

问题 2：CTC input_length 和实际输出时间步数不匹配

CTC Loss 要求 input_lengths 中的值不超过 log_probs 的时间步数 T。如果写错了（比如把图片宽度当成了时间步数），会报错：

正确做法：用 log_probs.size(0) 动态计算时间步数，而不是硬编码。

问题 3：CER 几乎不变，但 loss 一直在降

现象：训练 loss 从 15 降到 3，但 val CER 始终在 0.8 左右。

原因：模型学会输出”大概率是 blank”，从而 loss 降低（因为大部分时间步确实是 blank），但解码出来的文本还是乱码。

解决：

  • 检查字符集定义，确认训练字符集和标注字符集一致
  • 增大 batch_size，让模型接触到更多非 blank 样本
  • 降低初始学习率，避免模型过早走向”懒人解”（全 blank）

问题 4：验证集 CER 远低于训练集 Loss 对应的水平——模型过拟合

现象：train loss 继续降低，val CER 却不降反升。

解决手段从弱到强：

  1. 增大 weight_decay（L2 正则化）
  2. 增强数据增强（随机亮度、高斯噪声、弹性变形）
  3. 添加 Dropout 层（CNN 每层后加 nn.Dropout2d(0.1)）
  4. 减少模型参数（缩小 cnn_out / rnn_hidden）
  5. 增大训练集规模

小结

完整的训练流程可以概括为：数据加载 (OcrDataset + DataLoader) → 模型前向 (CRNN.forward) → CTC Loss 计算 → 反向传播 (loss.backward) → 参数更新 (optimizer.step) → 验证评估 (CER) → 调度器调整 (scheduler.step) → Early Stopping 判断 → 保存检查点。

CTC Loss 的精妙之处在于它让模型自己去学对齐——不需要人工标注每个字符的像素位置。这在前深度学习时代是不可想象的。
