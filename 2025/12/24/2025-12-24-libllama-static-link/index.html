天气与目标

十二月末，窗外飘着小雪，电脑风扇却转得嗡嗡响——编译 C++ 库的时候，CPU 从来不嫌冷。

上次我们把 llama.cpp 编成了 JNI 动态库 libllama-bridge.so，一笔带过了 libllama.a 的编译过程。但这一步其实暗坑密布：交叉编译工具链怎么配、CMake 的 IMPORTED 静态库怎么写、链接顺序为什么不能颠倒、编出来的 .a 怎么验证架构对不对。今天的任务就是把这一个环节彻底拆开，每个命令、每个参数、每个常见报错都讲清楚——让你能独立完成 llama.cpp 源码 → libllama.a → libllama-bridge.so 的完整链路。

原理

为什么用静态库而不是动态库

在 Android 上集成 llama.cpp，你其实有两个选择：

方案 产物 体积 加载方式 符号可见性
静态库 libllama.a 编入 libllama-bridge.so 一个 .so，约 15MB 一次 System.loadLibrary 符号内嵌，外部不可见
动态库 libllama.so 两个 .so 分开发布 两个文件，总和略大 需加载两次或设依赖 符号暴露，可能冲突

我们选静态库的理由有三：

  1. 分发简单——一个 libllama-bridge.so 包含了所有 llama.cpp 的代码，APK 里只多一个文件。
  2. 符号隔离——llama.cpp 的 ggml_* 内部函数不会污染全局符号表，避免和其他库冲突。
  3. 链接时优化——编译器在链接阶段能看到 llama.cpp 和 bridge 的全部代码，可以做跨模块内联和死代码消除。

代价是：libllama.a 不支持按需加载，必须重新编译 libllama-bridge.so 才能更新 llama.cpp。

交叉编译的本质

你在 Windows x86_64 上跑编译器，但生成的机器码要在 Android ARM64 手机上运行——这就是交叉编译。NDK 的 Clang 编译器通过 --target 参数指定目标三元组：

编译时 Clang 会用 NDK 内置的 sysroot（${NDK}/toolchains/llvm/prebuilt/windows-x86_64/sysroot），里面有 Android 的头文件和 stub 库，确保编出来的代码只调用 Android 上存在的系统函数。

可能遇到的问题

问题 1：undefined reference to ggml_* —— 链接顺序不对

原因：在 CMakeLists.txt 中，target_link_libraries(llama-bridge ${log-lib} llama) 把 libllama.a 放在了 llama-bridge 之前或中间。链接器从左到右处理：遇到 llama-bridge 中的未定义符号 ggml_init，它只会向后查找库。

解决：确保 llama 在 llama-bridge 之后：

问题 2：relocation cannot be used when making a shared object —— 忘了 -fPIC

原因：编译 .o 时没加 -fPIC。静态库编入动态库，.o 文件必须是位置无关的。

解决：重新编译所有 .o 文件，加上 -fPIC 标志。

问题 4：file format not recognized —— 用错了编译器

原因：用 x86_64 的 g++ 编译了 ggml.o，然后尝试用 NDK 的 llvm-ar 打包。.o 的格式和 llvm-ar 期望的架构不匹配。

解决：确认编译器路径正确——必须是 aarch64-linux-android26-clang++.cmd，不是系统自带的 g++。

问题 5：编出来的 .so 在设备上加载时崩溃，日志显示 SIGILL

原因：编译器生成的机器码包含设备 CPU 不支持的指令。通常因为目标 API Level 设得太高（如 android34 但设备是 Android 8.0），或者混入了 x86_64 的 .o。

解决：

  1. 降低 API Level：aarch64-linux-android26（对应 Android 8.0）
  2. 逐个检查 .o 文件的架构
  3. 用 llvm-readelf -A 查看具体启用了哪些 ARM 扩展

小结

一条 target_link_libraries(llama-bridge llama ${log-lib}) 背后，是交叉编译工具链配置、源文件逐个编译、符号验证、路径调试四个环节的配合。这篇文章把每个环节都拆开写了——从设置 $TARGET 变量到用 llvm-nm 检查符号，从 CMake 的路径解析到 .so 的依赖验证。

关键要点总结：

  1. **交叉编译不是”换个编译器”**——目标三元组决定了 sysroot、指令集、链接行为。
  2. -fPIC 不是可选的——编 .so 必须加。
  3. 链接顺序从左到右——被依赖的库放后面。
  4. 验证永远不嫌多——llvm-nm 看符号、llvm-readelf 看架构，花一分钟省一小时。

下一步：有了 libllama-bridge.so，Kotlin 侧怎么声明 external fun、JNI 函数签名怎么匹配——这是下一篇的内容。
