天气与目标

十二月中旬，冬至未至，暖气已开。这种天气最适合窝在显示器前做一件最不”开发”的开发工作——搭环境。

剪枝后的 Qwen 2.5 模型要跑到 Android 上，必须把 llama.cpp 编译成 arm64-v8a 的 .so 库。这意味着在 Windows x86_64 上交叉编译出 Android ARM64 的原生代码。今天的目标：用 NDK 在 Windows 上搭好交叉编译环境，把 llama.cpp 编成 libllama.a，再配好 JNI 桥接层的 CMakeLists.txt。

原理：什么是交叉编译？为什么这么麻烦？

交叉编译的本质

交叉编译 = 在一个架构的机器上，编译出另一个架构的二进制文件。这需要两样东西：

  1. **交叉编译器 (Cross Compiler)**——能在 x86_64 上跑、但输出 ARM64 机器码的 C/C++ 编译器。NDK 提供的 aarch64-linux-android26-clang 就是它。
  2. **目标平台的系统库 (Sysroot)**——Android 的 libc（Bionic）、libm、libdl 等。NDK 的 sysroot 目录里有这些。

第二步：NDK r26 vs r27——选哪个？

截至 2024 年底，NDK 有两个主流版本：

特性 NDK r26 NDK r27
Clang 版本 17.x 18.x
最低 API Level 21 24
std::filesystem 需要额外链接 libc++_shared 内置支持
libc++ 稳定版 ABI 稳定 ABI 可能调整
llama.cpp 兼容性 实测通过 偶有编译警告
社区使用量 主流，资料多 新，某些第三方库未适配

推荐 NDK r26，原因：

  1. llama.cpp 在 r26 上编译零警告零错误，r27 偶有 std::execution 相关报错。
  2. r26 设置最低 API 21，兼容 Android 5.0+，覆盖几乎所有还在用的设备。
  3. 社区资源和 StackOverflow 答案基本都是针对 r25/r26 的。

第三步：选择 ABI——为什么只用 arm64-v8a？

Android 支持四种 ABI（Application Binary Interface）：

ABI 架构 设备覆盖 备注
arm64-v8a ARM 64-bit ~95% (2017 年后) 我们只编译这个
armeabi-v7a ARM 32-bit ~5% (老旧设备) 不支持 ARM NEON 的量化和一些指令
x86_64 Intel/AMD 64-bit < 0.1% (模拟器) 开发测试用
x86 Intel 32-bit 几乎 0 已废弃

只编译 arm64-v8a 的理由：

  1. llama.cpp 在 32 位 ARM 上跑不动——1.5B 模型即使量化后也需 ~600MB 加载空间，32 位进程地址空间只有 3GB（用户态），加了系统开销根本不够。
  2. 维护成本——每多一个 ABI，编译时间翻倍，APK 体积翻倍，测试矩阵翻倍。我们不是商业 App，不追求 100% 设备覆盖。
  3. Google Play 要求——2021 年起强制 64 位。

第七步：配置 CMakeLists.txt——JNI 桥接层

app/src/main/cpp/CMakeLists.txt：这个文件告诉 Android Gradle Plugin 怎么编译 JNI 代码：

CMakeLists.txt 要点逐行解释：

  1. SHARED 类型——JNI 库必须编译为 .so（动态库）。System.loadLibrary("llama-bridge") 加载的是 libllama-bridge.so，不是 .a。
  2. IMPORTED 类型——libllama.a 是预编译产物，不在 CMake 源码树里。IMPORTED 告诉 CMake “这个库已经存在，别试图编译它”。
  3. **${ANDROID_ABI}**——由 Android Gradle Plugin 注入，值是 ndk.abiFilters 中指定的 ABI。我们设了 arm64-v8a，所以它总是 arm64-v8a。
  4. **${CMAKE_SOURCE_DIR}/../jniLibs/**——CMake 的工作目录是 app/src/main/cpp/，.. 就是 app/src/main/，jniLibs 在它下面。
  5. **${log-lib}**——Android 的系统日志库 liblog.so，编译时不嵌入，运行时由系统提供。

问题 6：编译成功但 Android 端加载模型 OOM

现象：llama_model_load_from_file() 返回 null，或加载后 App 闪退。

原因：即使 .so 编译正确，1.5B 模型的 GGUF 文件在加载后占 ~1.2GB 内存，加上推理时的 KV cache，总量可能超过设备可用内存。

小结

在 Windows 上搭 NDK 交叉编译环境，核心主线只有一条：

  1. NDK r26——llama.cpp 验证最稳定的版本，设 API 26 覆盖足够广。
  2. 只编译 arm64-v8a——32 位 ARM 跑不动大模型，x86 只存在于模拟器。
  3. 编译选项四要素——-O3（加速）、-fPIC（位置无关）、-static-libstdc++（免带 libc++_shared）、-DGGML_USE_LLAMAFILE（SIMD 优化）。
  4. 验证不能省——llvm-readelf 看架构、llvm-nm 看符号、解压 APK 看产物。

编译环境搭好了，llama.cpp 的代码在 Android 上跑起来就是”加一行 external fun“的事（虽然 JNI 桥接层还需要下一篇来写）。交叉编译这个坎，过了之后就是一片坦途——至少编译链不会再给你添麻烦了。
