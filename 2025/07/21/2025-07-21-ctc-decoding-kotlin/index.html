天气与目标

七月的深圳，午后暴雨来得快去得也快，空气里混着湿热和泥土味。

前几篇我们走通了模型推理的完整链路：Bitmap 预处理、TFLite Interpreter 初始化、CRNN 模型导出。但推理出来的东西——一个 (64, 41) 的浮点数矩阵——距离人类可读的”一行字幕文本”还差最后一步：CTC 解码。

今天的目标：把 CTC 贪心解码算法从 Python 原封不动地重写到 Kotlin，弄清楚为什么必须自己写而不用 TFLite 内置算子，把字符集映射、大小写处理、blank token 的坑一个一个踩平。

一、为什么不直接用 TFLite 内置的 CTC 解码？

1.1 理想情况：模型自带解码 Op

在 Python 训练侧，CTC 解码只不过是几行代码：

PyTorch 的 ctc_decode 内部做了 beam search 和 blank 合并，一行代码出结果。你甚至可以把解码逻辑写进 ONNX 导出图的最后一个节点——只要导出的算子链条是完整可微的。

1.2 现实：TFLite 没有 CTC Decoder Op

转换到 TFLite 时，问题来了。打开 TensorFlow Lite 的算子支持列表，你会发现：

  • CTCGreedyDecoder —— 不存在
  • CTCBeamSearchDecoder —— 不存在
  • 连 ArgMax 在旧版本 TFLite 中也是通过 BuiltinOperator_ARG_MAX 支持的，和 CTC 没有关系

也就是说，CTC 解码从来不是一个 TFLite Builtin Op。你的 CRNN 模型在导出为 tflite 时，最后输出的只是一个 (1, 64, 41) 的 logits（或 softmax 概率），没有解码步骤。解码必须在模型外部自己实现。

1.3 如果强行把解码塞进模型呢？

你可能会想到一个折中方案：在导出 ONNX 时把 ArgMax + 去重逻辑写成原生算子链，期望 TFLite 转换器能理解。但这条路基本走不通，原因是：

  1. 去重（dedup）和 blank 合并是带状态的操作——不是逐元素张量变换，而是需要跨时间步维护一个”上一个字符”状态。这不是标准张量算子的语义范围。
  2. **即使 TFLite 支持了 ArgMax**，后面紧跟的非张量操作（有条件跳过、状态比较）也无法表示为 DAG 中的节点。
  3. 强行塞入模型会增加模型复杂度，且转换工具链（onnx2tf / onnx-tensorflow）大概率报错或产生错误结果。

结论：CTC 解码必须自己在应用层手写。 好消息是，Greedy Decoding 的算法量小到只有十几行，完全不像”造轮子”——更像是”把训练侧本来就应该分离的后处理逻辑补上”。

五、大小写处理

5.1 模型只输出小写

CRNN 模型的训练集通常只包含小写字母——原因很简单：把 Hello 和 hello 视为两个不同的类会翻倍字母类别的数量，且无实际语义区分。所以训练时的字符集统一为小写，模型也只学会输出小写。

但电影字幕很可能是混合大小写的，比如 "I'm sorry"、"STOP!"、"Mr. Smith"。如果直接把模型输出的全小写字符串跟字幕原文做对比（比如字幕遮挡判断、时间轴匹配），会全部对不上。

六、常见 Bug 与踩坑记录

Bug 1：忘了 trim()，结果以空格开头/结尾

现象：OCR 输出正常，但和字幕做精确匹配时永远返回 false。

原因：模型的时间步序列两端总是有大量 blank token。解码过程中，开头的连续 blank 不输出，但如果中间有空白符被误识别为空格，结果可能以空格开头。或者更常见的情况是——StringBuilder 本身没有以空格开头，但最后一个有效字符后面有些空白时间步输出了空格。

修复：

注意 Kotlin 的 trim() 去掉的是 \u0020（普通空格），不会去掉 \u00A0（不间断空格）等。如果你的模型字符集里有特殊空白字符，需要自己写 trim 逻辑。

Bug 2：blank 索引差 1——解码全变乱码

现象：解码出来要么是空白字符，要么完全对不上输入。

原因：训练侧 CTCLoss(blank=0)，但解码时写成 blank=alphabet.length。

诊断方法：打印每个时间步的 argmax 索引和对应的字符：

如果某个时间步的 max 概率始终落在某个固定索引（比如索引 0），而这个索引对应的字符根本不出现——基本就是 blank 索引配错了。

Bug 3：空字符串处理——IntArray 越界

现象：ArrayIndexOutOfBoundsException 或返回空字符串导致下游逻辑崩溃。

原因：模型输出的 64 个时间步全部判为 blank（比如传了一张全黑/全白的无效图像），解码得到空字符串。下游如果用 result[0] 做字符访问就会 crash。

八、性能与扩展

8.1 Greedy vs Beam Search

本文实现的 Greedy Decoding（贪心解码）是 CTC 解码中最简单的变体。还有另一种是 Beam Search Decoding：

方法 时间复杂度 准确性 实现复杂度 适用场景
Greedy O(T × C) 一般（单字最优未必全句最优） 极低 简单字幕、数字识别
Beam Search O(T × C × beam_width) 更高 中等 长句、手写体

对于电影字幕——每个字幕框通常只有 3-15 个单词，且字体清晰、背景干净——Greedy Decoding 的准确率已经足够。Beam Search 的收益（提升 1-3% 字符准确率）不值得付出 5-10x 的计算开销。
