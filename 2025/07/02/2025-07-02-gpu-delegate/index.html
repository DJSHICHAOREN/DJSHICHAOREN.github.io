GPU Delegate 加速与 CPU 回退策略：TFLite 推理的性能博弈

天气与目标

七月初，手机都开始发烫了。

OCR 识别每次推理需要 50-150ms（CPU），但我们的目标是 < 50ms。今天研究 TFLite 的 GPU 加速方案，以及 GPU 不可用时如何优雅回退到 CPU。

原理

TFLite 的三种推理后端

后端 加速原理 延迟 兼容性
CPU (XNNPACK) ARM NEON 指令集 50-150ms 所有设备
GPU Delegate OpenGL ES Compute Shader 15-40ms Android 4.1+
NNAPI 芯片专用 NPU/DSP 5-30ms Android 8.1+，但实现差异大

为什么需要回退策略

GPU Delegate 不是在所有设备上都可用：

  1. 某些低端手机 GPU 不支持 Compute Shader
  2. GPU 内存不足时初始化会失败
  3. 某些 GPU 驱动有 bug，特定 op 会算错
  4. 模拟器通常没有 GPU 加速

所以必须实现：优先 GPU → GPU 失败则 CPU → CPU 兜底一定有。

遇到的问题

问题 1：UnsatisfiedLinkError: No implementation found for...

原因：APK 中缺少 GPU Delegate 的 native 库。

解决：abiFilters 必须包含 GPU Delegate 支持的架构（arm64-v8a）。如果只用 armeabi-v7a，GPU Delegate 不支持。

问题 2：GPU Delegate 初始化成功但推理结果全是 0

原因：GPU 和 CPU 的浮点精度差异导致某些 op 输出异常（尤其是 INT8 量化模型在 GPU 上运行）。

解决：

  • 用 FP16 模型跑 GPU（不要用 INT8 + GPU 的组合）
  • 或在 GPU Delegate 初始化时指定 GpuDelegate.Options().setPrecisionLossAllowed(true)

问题 3：模拟器上 GPU Delegate 崩溃

原因：Android 模拟器的 GPU 实现（SwiftShader）不支持 Compute Shader 的某些扩展。

解决：在模拟器上直接用 CPU，用 isGpuCompatible() 自动检测会返回 false。

问题 4：GPU 推理的 first run 特别慢

原因：GPU Delegate 在第一次推理时会编译 shader，这个开销可达 200-500ms。

解决：在应用启动时跑一次空推理（warmup）：

小结

TFLite 的 GPU Delegate 可以将 OCR 推理从 100ms 降到 20ms——这在实际使用中是”卡不卡”的差别。但 GPU 不是银弹——兼容性检测、回退策略、warmup 缺一不可。
