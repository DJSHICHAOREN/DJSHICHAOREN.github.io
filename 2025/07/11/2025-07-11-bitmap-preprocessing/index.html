Bitmap 到 Tensor：Android 上的图像预处理与零拷贝优化

天气与目标

七月中的午后，窗外知了声此起彼伏。

今天的目标听起来很简单：把一张 Android Bitmap 转成 OCR 模型要吃的 ByteBuffer Tensor。 但实际上要做好这件事（正确的灰度计算、正确的归一化范围、避免额外的内存分配），每一步都有坑。

关键决策点

问题 选项 推荐
灰度公式 简单平均 (R+G+B)/3 vs 加权 0.299R+0.587G+0.114B 加权（人眼对不同颜色敏感度不同）
归一化范围 [0, 1] vs [-1, 1] 与训练时一致（本文用 [-1,1]）
内存分配 每帧新建 ByteBuffer vs 复用 复用（减少 GC 压力）
Bitmap 格式 ARGB_8888 vs RGB_565 ARGB_8888（最通用）

可能遇到的问题

问题 1：归一化范围错误——OCR 输出全是乱码

原因：训练时用了 (x - 0.5) / 0.5（范围 [-1,1]），部署时忘了归一化。

检查方法：打印几个输入值，确认范围。

问题 2：ByteBuffer 的 position 不对导致推理失败

TFLite 的 Interpreter.run() 从 ByteBuffer 的 position 开始读，读 remaining() 个字节。如果忘了 rewind() 或 flip()，读到的是空数据或上一次的数据。

调试：Log.d("Buf", "pos=${buffer.position()}, limit=${buffer.limit()}, remaining=${buffer.remaining()}")

问题 3：Bitmap.getPixels 返回的是预乘 Alpha（premultiplied alpha）

Android Bitmap 在特定情况下会自动预乘 alpha——RGB 值 = 原始RGB × Alpha/255。这会影响灰度计算。

解决：确保 Bitmap 是没有预乘 alpha 的。Bitmap.Config.ARGB_8888 默认不预乘，但如果从某些 Decode 选项创建的 Bitmap 可能是预乘的。

性能对比

方法 每帧耗时 内存分配 GC 压力
基础版（每次 new） 3-5ms ~120KB/frame 高
复用缓冲区版 1-2ms 0 无
Canvas 缩放（代替 createScaledBitmap） 快约 30% 0 无

小结

图像预处理看起来是”几行代码”，但做到零分配零拷贝需要仔细设计缓冲区的生命周期。对于 OCR 这种每帧都要推理的应用，预处理每省 1ms，就是每秒多处理 2-3 帧的差距。
