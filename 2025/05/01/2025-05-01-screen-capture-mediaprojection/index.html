用 MediaProjection 捕获屏幕画面，裁剪字幕区域

天气与目标

五一是劳动者的节日。今天也来”劳动”一下——让悬浮窗能看到手机屏幕上的内容。

上一篇做好的悬浮窗只是一个空壳，现在需要给它装上”眼睛”——通过 Android 的 MediaProjection API 捕获屏幕画面，并裁剪出底部字幕区域，送给 OCR 引擎识别。

原理

MediaProjection 工作流程

为什么用 ImageReader 而不是 MediaRecorder？

  • MediaRecorder 是录视频用的，拿不到单帧
  • ImageReader 可以逐帧获取，适合 OCR 场景（不需要实时帧率，2-3 FPS 就够了）

裁剪策略

电影字幕通常位于屏幕底部 1/5 区域。把这个区域裁出来，OCR 只需要处理约 200×50 像素的图像（而不是 1080×2400 全屏），大大减少推理计算量。

可能遇到的问题

问题 1：ImageReader 缓冲区耗尽 → IllegalStateException: maxImages (2) already acquired

原因：acquireLatestImage() 取出的 Image 没有调用 close()，导致缓冲区的两个槽位被占满。

解决：**每次取完必须在 finally 中 close()**，这点非常容易忘。

问题 2：Bitmap 中有绿色/紫色条纹（Stride 问题）

原因：ImageReader 的 rowStride 可能大于 width * 4（GPU 会对齐到 64 字节等）。

解决：先用 width + padding 创建 Bitmap 再裁剪到实际宽度。上面的 imageToBitmap 方法已经处理了这个情况。

问题 3：MediaProjection 权限每次都要弹窗

原因：Android 设计如此——为了安全，MediaProjection 每次都要用户确认。

缓解：可以把 resultCode 和 data 保存下来，在应用生命周期内复用。但跨进程重启后就失效了。

问题 4：屏幕录制导致性能下降

原因：MediaProjection 捕获全屏帧（1080p 甚至 4K），每帧转 Bitmap 开销大。

优化方向：

  • 降低采样率（1 秒 1 帧，字幕变化不快）
  • 用小尺寸 ImageReader（但 VirtualDisplay 必须匹配分辨率）
  • 直接在 YUV 层做裁剪（不需要先转 RGB）

小结

现在悬浮窗 Service 能拿到屏幕的实时画面，并裁剪出字幕区域。下一步就是把这个裁剪后的 Bitmap 送给 TFLite 做 OCR 识别。

但在那之前，我们需要先有一个训练好的 OCR 模型，以及把它从 PyTorch 格式转成 Android 能加载的 .tflite 格式。
