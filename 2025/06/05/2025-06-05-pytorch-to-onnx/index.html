天气与目标

六月了，窗外下着雨，正是适合宅在电脑前折腾模型的好天气。

上一篇我们实现了 CRNN 模型——CNN + BiLSTM + CTC 三段式架构，在 PyTorch 上跑通了训练。但真正要用到 Android 端，必须把它导出为 ONNX，再转成 TFLite。听起来就是一行 torch.onnx.export() 的事——实际上，序列模型的导出暗藏玄机。

今天的目标：把 CRNN 模型从 PyTorch 导出为 ONNX，解决可变宽度、opset 兼容性、LSTM 算子的坑，最终用 onnxruntime 验证结果一致性。

背景：为什么需要 ONNX？

我们的部署管线是：

ONNX（Open Neural Network Exchange）是跨框架的中间表示。它重要的原因是：

  1. 解耦训练和部署框架：训练用 PyTorch（灵活、生态好），部署可能是 ONNX Runtime / TensorFlow Lite / OpenVINO / TensorRT。
  2. 图优化：ONNX 社区有大量算子融合、常量折叠的优化工具，模型跑起来更快。
  3. 验证中间状态：导出后立即用 onnxruntime 跑一遍，确认结果跟 PyTorch 一致，比直接转 TFLite 出错了摸不着头脑强得多。

但 ONNX 也有它的约束——它是一个静态图格式，而 PyTorch 是动态图。把动态的 Python 逻辑翻译成静态的图 IR，过程中很多东西会”丢”。

第二步：动态轴——处理可变宽度

为什么需要动态轴？

CRNN 模型输入是 (B, C, H, W) 形状的图片，其中：

  • B = batch size（批量大小）
  • C = 1（灰度单通道）
  • H = 32（固定高度，数据集 resize 时规范化了）
  • W = 可变（不同图片宽度不同，短的 “ok” 可能只有 30 像素宽，长的 “hello world” 可能 300 像素宽）

如果导出时不声明动态轴，ONNX 会把输入形状”烧死”为 dummy_input 的形状 (1, 1, 32, 200)。这样一来，部署时就只能处理宽度恰好 200 像素的图——完全没法用。

dynamic_axes 的写法

这里的数字是维度索引（0-based）。PyTorch 的 tensor 形状是 NCHW 布局：

CNN 输出把 H 压缩为 1 后，形状变成 (B, 512, 1, W')，经 squeeze 和 permute 后进入 LSTM 的是 (B, W', 512)。LSTM 输出 (B, W', hidden*2)，最后线性层映射为 (B, W', num_classes)。所以输出 shape 的 axis=1（序列长度，即 W'）也是跟着输入宽度变化的。

关键规则：声明了 {3: "width"} 之后，其他动态轴只要也用名字 "width" 引用，不必显式写出映射。但 sequence_length 不能叫 width（含义不同），所以要单独声明。

第三步：opset 版本——13 vs 17 怎么选？

opset_version 决定了 ONNX 算子集的版本。版本越高，支持的算子越多，但部署端的 ONNX Runtime 或转换工具可能不兼容。

opset 13（我们的选择）

我们代码里用的是 opset_version=13。这个版本的算子支持情况：

算子 opset 13 支持 备注
Conv 完全支持 -
BatchNorm 完全支持 -
ReLU 完全支持 -
MaxPool 完全支持 静态 kernel/stride
LSTM 支持但有限制 后面细说
Linear (Gemm) 完全支持 -
Squeeze / Unsqueeze 支持 (opset 13+) opset 13 起输入为 tensor
Reshape 支持 (opset 13+) opset 13 起支持 allowzero

opset 13 的兼容性最广：ONNX Runtime 1.5+、TFLite Converter、大部分边缘设备推理引擎都支持。

opset 17 有什么变化？

特性 opset 13 opset 17
LayerNorm 不支持 原生支持 (onnx.LayerNormalization)
GridSample 不支持 自带支持
新版 Reduce 算子 旧版 ReduceMean 等 统一为 Reduce* (17)
LSTM 双向 layout batch_first + bidirectional 同 opset 13 兼容

对 CRNN 来说，opset 13 和 17 的差异不大——我们没有 LayerNorm 也没有 GridSample。但 opset 17 导出时 PyTorch 的 trace 行为可能略有不同（主要是常量折叠策略），所以保持跟参考代码一致的 opset 13 更稳妥。

实际操作：测试两个 opset

对 CRNN，差异通常是 0 或极少。如果 opset 17 导出了 LayerNormalization 而你下游的 TFLite Converter 不认识它，那就退回到 13。

第四步：算子的坑——哪些算子能导出，哪些不能？

这是最容易被忽略、也是最让人抓狂的部分。PyTorch 的 torch.onnx.export 不是”万能翻译器”——它只能翻译那些在 ONNX 标准中有对应实现的算子。

CRNN 模型的算子盘点

能正常导出的算子

  • Conv2d、BatchNorm2d、ReLU、MaxPool2d：这些是 ONNX 一级公民，从 opset 9 开始就全面支持。导出没有任何问题。
  • Linear（全连接层）：ONNX 中用 Gemm 算子表示，导出无压力。
  • Squeeze、Permute（Transpose）：ONNX 中有 Squeeze 和 Transpose 算子，导出正常。

LSTM：能导出但有坑

PyTorch 的 nn.LSTM 可以导出到 ONNX，但 ONNX 的 LSTM 算子跟 PyTorch 的实现有几处不同：

坑 1：batch_first 参数

PyTorch 的 LSTM 支持 batch_first=True（输入 (B, T, C)），但 ONNX 的 LSTM 算子默认输入是 (T, B, C)（seq_length first）。

幸运的是，PyTorch 2.0+ 的 exporter 会自动插入 Transpose 算子来解决这个差异——导出时不需要手动调整。但如果你用的是旧版 PyTorch（1.x），可能需要在导出前手动 transpose。

坑 2：双向 LSTM 的 layout

ONNX 的 LSTM 算子通过 direction="bidirectional" 属性支持双向。但 ONNX 中双向 LSTM 的 Y 输出形状跟 PyTorch 略有不同：

  • PyTorch bidirectional：Y 的最后一维是 hidden_size * 2（正反向拼接）
  • ONNX bidirectional：默认情况下 Y 的处理方式跟 PyTorch 2.x 的 exporter 对齐了

我们用 batch_first=True, bidirectional=True 导出的模型，在 onnxruntime 上验证结果跟 PyTorch 一致——前提是 PyTorch 版本 >= 2.0。

坑 3：LSTM 的 h 和 c 输出

torch.onnx.export 默认用 trace 模式：实际跑一遍 forward，记录走过的算子。对 LSTM，trace 模式下 (h_n, c_n) 这两个返回值如果被丢弃（rnn_out, _ = self.rnn(features)），导出图中就不会包含它们——这正好是我们需要的，因为 CTC decoder 不依赖隐状态。

CTC：根本不用导出！

CTC Loss 只在训练阶段使用。我们导出的 ONNX 模型只需要输出 (B, T, num_classes) 的 logits（字符概率分布），CTC 贪心解码或 Beam Search 解码在 ONNX 图之外用 Python / C++ / Java 实现。

这一点很重要：不要试图在 ONNX 里实现 CTC 解码。CTC 解码包含 argmax、循环去重、移除 blank 等操作，虽然 in theory 可以用 ONNX 算子拼出来，但图会变得极其复杂，而且每个平台还得重复实现一遍。正确的做法是：

小结

从 PyTorch 导出 ONNX 看起来是一行代码的事，但序列模型（CRNN、Transformer、语音模型等）有几个必须处理的点：

  1. 动态轴：宽度可变，input 和 output 都要声明，不然部署时只能处理一个尺寸。
  2. opset 版本：不是越新越好，看你的下游工具链支持什么。CRNN 用 opset 13 兼容性最广。
  3. 算子兼容性：Conv/BN/ReLU 随便导；LSTM 能导但注意 PyTorch 版本；CTC 不在模型里导出，解码在模型外做。
  4. 验证是刚需：导出完必须用 onnxruntime 跑一遍，各种宽度都测，不能只信 export() 没报错。
