1.3 Interpreter 构造时做了什么

翻开 TFLite 源码（org.tensorflow.lite.Interpreter），构造阶段的工作包括：

  1. 解析 FlatBuffer 模型图——遍历算子节点，构建计算图拓扑
  2. 内存分配——根据 input/output tensor 的形状预先分配缓冲区
  3. 算子注册与委托匹配——检查哪些算子可以用 GPU/NNAPI 委托加速
  4. 线程池初始化——除非显式指定线程数，否则默认 Runtime.getRuntime().availableProcessors() 个线程

这些操作是 IO 密集 + CPU 密集 的混合，不该每帧执行。

三、多线程崩溃：同一个 Interpreter 不能跨线程共享

3.1 现象

假设你做了两个功能：一个悬浮窗实时识别，一个手动截图识别按钮。用户点了按钮，但悬浮窗还在后台跑——两个线程同时调 interpreter.run()。

3.2 根本原因

TFLite 的 Interpreter 不是线程安全的。官方文档写得清清楚楚：

Interpreter is not thread-safe. If you need to invoke the same interpreter from multiple threads, you must provide your own synchronization.

内部原因不难理解：run() 方法会修改 Interpreter 内部的 tensor 状态、内存分配器状态、算子执行上下文。两个线程同时写这些共享状态，轻则算错结果，重则 native 层内存越界 crash。

3.3 解决方案：对象池 / 线程局部 Interpreter

方案 A：synchronized 加锁（简单但粗暴）

适用场景：推理耗时在 20-50ms 以内，调用频率不高（比如手动截图），加锁造成的等待可以忽略。

我们的场景是电影字幕识别——帧率通常 0.5-2fps（不需要每帧都识别），synchronized 加锁完全够用。这正对应了 OcrEngine.kt 里的设计：单例 + 不在 recognize 上加锁，由上层调用方保证串行调用。

四、MappedByteBuffer vs ByteBuffer：模型加载方式的差距

4.1 两种加载方式

4.2 核心区别

MappedByteBuffer 的优势：

  1. 延迟加载：只有被访问的页才会加载到物理内存。5MB 模型文件不等于占用 5MB RAM
  2. 零拷贝：native 层（TensorFlow Lite C++ 核心）通过指针直接访问映射区域，不需要 JNI 层再做一次数据拷贝
  3. 系统级缓存复用：如果模型文件在同一进程内被多次映射，内核的 Page Cache 可以避免重复 IO

五、内存管理：Interpreter 的 native 内存与 GC

5.1 Interpreter 的内存模型

关键问题：Java 侧的 Interpreter 对象很小（几十字节），但 native 侧的内存可能是几 MB 到几十 MB。GC 只管理 Java 堆，不知道 native 侧有多大。 如果你依赖 GC 来回收 Interpreter（即不主动调用 close()），可能会在 GC 触发前一直持有大量 native 内存。

七、实战总结：电影英语助手的 OcrEngine 设计回顾

回头看 OcrEngine.kt 的最终设计，它没有用对象池，也没有用专属线程——因为我们的场景是每隔 0.5-1 秒做一次字幕区域 OCR，频率很低：

几个设计要点：

  1. Interpreter 在 init 块中创建——随 OcrEngine 对象一起初始化，全局只建一次
  2. **loadModelFile 使用 MappedByteBuffer**——零拷贝，比 ByteBuffer 快约 60%
  3. **recognize 不加 synchronized**——调用方自行保证串行，避免不必要的锁开销
  4. close() 暴露给外部——在 Application.onTerminate() 中调用，释放 native 内存

如果未来需要多线程并发识别，只需要在调用方加一层 synchronized 或用 ConcurrentLinkedQueue 排队即可，OcrEngine 本身的代码不用动。
