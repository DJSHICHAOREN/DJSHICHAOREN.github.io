天气与目标

二月初，立春将至，但手机的后盖比春天先”热”了起来。

上一篇我们调好了 Q4_K_M 量化模型的推理速度——11 tok/s，看电影时后台默默翻译字幕，用起来挺舒服。但舒服只持续了大约 10 分钟——然后手机开始发烫，推理速度从 11 tok/s 断崖式跌到了 3 tok/s，翻译一个字幕要等 5 秒。用户关掉了 App。

今天的目标：搞清楚为什么手机跑 LLM 会发热降频，以及我们能做什么来让推理持续、稳定、不烫手。

原理

发热的数学本质

手机是一个密闭的被动散热系统。CPU/GPU 消耗的电能几乎 100% 转化为热能，而散热能力受限于机身表面积：

LLM 推理为什么特别热

对比不同任务的功耗特征：

任务 计算模式 CPU 占用 GPU 占用 持续时长 发热程度
UI 渲染 间歇性 ~5% ~10% 按需 无感
OCR 识别 单次短跑 ~30% ~25% 0.05s/次 轻微
视频播放 硬件解码 ~10% ~20% 持续 温和
LLM 推理 持续马拉松 ~85% ~65% 连续几分钟 严重

LLM 推理的特殊之处在于：

  1. 计算密集且持续——每生成一个 token 就要跑一遍整个神经网络
  2. 没有喘息机会——字幕一句接一句，推理不间断
  3. GPU 和 CPU 同时满载——n_gpu_layers=16 意味着两边都在高负荷工作

遇到的问题

问题 1：THERMAL_STATUS_* 权限不足

解决：addThermalStatusListener 不需要额外权限，但某些 OEM 的自定义 ThermalManager API 需要 DEVICE_POWER 签名权限（普通应用无法获取）。所以只用 PowerManager.currentThermalStatus——它是公开 API，不需要权限。

问题 2：间歇策略导致翻译延迟

如果跑 5 句歇 3 秒，而第 6 句字幕刚好出现在休息期间，用户会感觉”卡住了”。

解决：把shouldRest()的触发改为非强制——如果队列中有新的字幕还没处理且用户正在看字幕区域，跳过这次休息：

小结

移动端跑 LLM，性能瓶颈不在模型大小——在散热。再好的量化、再快的 GPU，跑三分钟后手机一热，全都白费。

核心生存法则就两条：

  1. 制造间隙——不要持续满载，跑一段歇一段，制造散热窗口
  2. 提前干活——利用息屏、空闲、深夜的时间预处理，把”热”从用户体验的时间线中抹掉

实测数据证明了这一点：间歇式热管理让 120 分钟电影全程保持 10 tok/s，而连续推理在第 30 分钟就崩了。再加息屏预处理，设备温度全程不超过 38°C，用户甚至感觉不到后台有 LLM 在跑。
