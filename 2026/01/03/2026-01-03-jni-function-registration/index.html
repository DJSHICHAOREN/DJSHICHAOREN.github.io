天气与目标

元旦刚过，天气冷得让人只想缩在被窝里写代码。今天不下雪，但寒风把窗外的树枝吹得簌簌响——有点像 JNI 函数签名配错时终端里滚过的 error log，刺耳又密集。

上一篇文章我们把 libllama.a 静态链接到了 libllama-bridge.so。库是有了，但 Kotlin 代码和 C++ 代码之间还隔着一道 JNI 的”翻译墙”——Kotlin 写 external fun nativeInit(path: String, nThreads: Int): Long，C++ 侧怎么接住这个调用？String 在内存里是什么？Long 变成 C++ 的什么类型？函数名为什么那么长？今天的目标是把这堵墙拆掉——从 JNI 命名规则到类型映射表，从 GetStringUTFChars 的编码开销到 RegisterNatives 的手动注册，逐个讲透彻。

原理

JNI 调用全链路

Kotlin 在虚拟机里运行，C++ 在原生堆上跑数据。JNI 就是这两者之间的翻译官：

每次 nativeInit() 调用，经历了以下步骤：

  1. Kotlin 调用 nativeInit → ART 虚拟机查找 libllama-bridge.so 中名为 Java_com_movieenglish_assistant_llama_LlamaEngine_nativeInit 的函数。
  2. 参数转换 → String 转为 jstring（一个指向 JVM 堆中对象的句柄）、Int 转为 jint（直接值拷贝）。
  3. 进入 C++ → 函数体执行，GetStringUTFChars 从 jstring 取出 const char*。
  4. 返回值转换 → jlong 回到 Kotlin 侧变成 Long。

两种注册方式：自动 vs 手动

方式 原理 优点 缺点
自动注册（函数命名约定） 函数名 = Java_包名_类名_方法名，ART 通过 dlsym 按名字查找 零配置，改了 Kotlin 函数名自然会报错 函数名冗长、不能改名或混淆
手动注册（RegisterNatives） 在 JNI_OnLoad 中调用 env->RegisterNatives，显式绑定函数指针 函数名可以任意、启动时一次性绑定、支持动态映射 多写一段注册代码，出错时定位更难

我们目前用的是自动注册——简单、直接、和代码结构一一对应。对于只有三个 JNI 函数的项目来说，手动注册的好处（缩短函数名、代码混淆）还不值得那额外的代码量。

第三步：String 的编码陷阱

jstring 到 const char* 的转换看似简单，实际有编码问题：

GetStringUTFChars vs GetStringChars：

函数 编码 返回类型 用途
GetStringUTFChars Modified UTF-8 const char* C/C++ 世界标准字符串操作
GetStringChars UTF-16 const jchar* 需要和 Java 内部编码一致时

为什么我们选 UTF-8？ llama.cpp 的 llama_tokenize API 接收 const char*，UTF-8 直接兼容。如果用了 GetStringChars 拿到 UTF-16，还需要先转码，多一次内存分配。

Modified UTF-8 的坑：

  • \0（null 字符）被编码为 \xC0\x80，不是真正的 \0
  • 代理对（surrogate pairs）的编码方式不同
  • 对大多数 ASCII/中文文本，和标准 UTF-8 相同

小结

JNI 函数注册本质上是一道命名约定——Kotlin 侧声明 external fun，C++ 侧用对应的 Java_包名_类名_方法名 函数接住。中间的参数翻译靠 JNI 类型映射表：String 变 jstring 再 GetStringUTFChars，Long 直接当 jlong 用，Int 直接当 jint 用。

最容易出问题的地方永远是字符串——GetStringUTFChars / ReleaseStringUTFChars 的配对、编码差异、释放后使用。编译器和链接器不检查这些运行时约定，只能靠代码纪律和仔细排查。

三个 JNI 函数，对应三个生命周期阶段——init、generate、free。这三个阶段构成了 LlamaEngine 和 llama.cpp 之间的完整对话协议。下一篇，我们聚焦在 nativePtr 这个 Long 值上：它在 Kotlin 和 C++ 之间传递了什么、怎么防止 double-free、以及 GGUF 模型文件怎么从 assets 搬出来。
