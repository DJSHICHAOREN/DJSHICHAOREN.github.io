天气与目标

一月中旬，窗外干冷的北风把行人都吹得缩了脖子。看着代码里 nativePtr 那个 Long 变量——它是 C++ 堆上一块内存的唯一凭证，弄丢了就是内存泄漏，弄重了就是 double-free。冷风是提醒，内存管理更是提醒——每一步都要精确。

前两篇我们打通了静态库编译和 JNI 函数签名。今天聚焦在那个在 Kotlin 和 C++ 之间来回传递的 Long 值上：GGUF 模型怎么从 APK 的 assets 里搬出来、LlamaContext* 的生命周期怎么管理、什么时候该释放、怎么防止重复释放、以及内存泄漏和 double-free 的调试手段。

原理

为什么 assets 里的文件不能直接 fopen

Android APK 是个 ZIP 压缩包。assets/ 目录下的文件是包内的压缩条目，而不是文件系统上的独立文件。C/C++ 代码的 fopen、std::ifstream、llama.cpp 的 llama_model_load_from_file 都期望一个文件系统路径，它们内部是直接走内核的 open() 系统调用——无法穿透 ZIP 容器。

Native 指针作为不透明句柄

为什么用一个 Long 而不是把 C++ 对象的字段一一映射到 Kotlin？

这叫不透明指针模式（Opaque Pointer Pattern），也叫 PIMPL 的外族亲戚。好处：

  1. Kotlin 零知识——Kotlin 侧不需要知道 LlamaContext 的 layout，C++ 可以随时改结构体字段。
  2. 边界清晰——所有权完全在 C++ 侧，Kotlin 只是”借”了一个数字。
  3. 跨语言安全——如果直接把结构体字段映射为 Kotlin data class，JNI 传输需要逐字段拷贝，字段多了容易出错。

代价是：nativePtr 的值本身没有任何保护——传一个错误的数字（比如 0xDEADBEEF）给 nativeFree，C++ 侧会直接崩溃。

第三步：使用中的句柄校验

为什么不在 init 里自动调用 init？

init() 是耗时操作（5-30 秒加载模型），generate() 被调用的场景可能对延迟敏感（用户实时查词）。分开两个方法，让调用方（通常是 ViewModel 或 Service）在启动阶段就提前调用 init()，等用户真正用的时候模型已经加载好了。

小结

nativePtr 这个 Long 值，是 Kotlin 世界通往 C++ 世界唯一的钥匙。它的生命周期很简单——从 nativeInit 的返回值获得，在 nativeFree 中归还。但管理的细节决定了你的 App 是优雅运行还是无声崩溃：

  1. GGUF 从 assets 拷贝到内部存储——因为 APK 是压缩包，C/C++ 不能直接打开。
  2. nativePtr == 0L 作为未初始化标志——0 在 C++ 侧不会返回（堆地址永远不是 0），是天然的哨兵值。
  3. **close() 后 nativePtr = 0**——防止 double-free 的最简单也是最可靠的方法。
  4. 释放顺序不能颠倒——ctx 依赖 model，model 依赖 backend，所以 ctx → model → backend。

这三篇连起来——静态库编译、JNI 函数签名、Native 指针生命周期——构成了我们”电影英语助手”中 llama 集成的完整拼图。下一步，是用 Kotlin 协程把这些封装成异步对话服务，让模型在后台默默推理的同时，主线程的 UI 依然丝滑。
