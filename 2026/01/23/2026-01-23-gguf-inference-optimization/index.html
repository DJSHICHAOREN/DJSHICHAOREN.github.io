GGUF Q4 量化模型的推理速度与内存优化

今日目标

一月下旬，上海终于有了点冬天的样子——室外 12 度，手机总算不烫手了。正好趁这个”凉爽期”把推理性能的底摸清楚。

上一篇我们把 llama.cpp 编译成了 JNI 库，模型能在手机上跑了。但”能跑”和”跑得好”之间隔着一整条优化链。今天的目标：系统性地测一遍 Q4_K_M 量化模型的推理速度和内存占用，找到最优的 n_gpu_layers、n_ctx、batch_size 配置组合。

原理

GGUF 量化类型对比

GGUF 格式支持多种量化精度，核心思路是把 FP32/FP16 的浮点权重映射到低位宽的整数空间：

量化类型 位宽 体积（1.5B） 质量损失 推理速度 适用场景
FP16 16bit ~2.8GB 无 慢（需 GPU） 服务端
Q8_0 8bit ~1.4GB 极小 中等 高端手机
Q5_K_M 5bit ~1.0GB 很小 快 中端手机
Q4_K_M 4bit ~900MB 可接受 快 目标场景
Q3_K_M 3bit ~750MB 明显 很快 低端手机
IQ2_XXS 2bit ~500MB 较大 最快 极限省内存

“K” 表示量化算法类型（k-quant），”M” 表示中等大小（medium），”S” 表示小（small）。K-quant 的核心改进在于：对模型中重要的权重层（如 attention 的 Q/K/V 投影）分配更高的位宽，对不那么重要的层分配更低位宽——混合精度量化，而非一刀切。

K-quant 的内部结构（以 Q4_K_M 为例）：

这个设计非常聪明——attention 的权重决定了模型”注意什么”，一旦量化损失过大，整个生成质量会崩溃。而 feed-forward 的权重更多是”记忆型”的，4bit 足够。

为什么不全放 GPU？ 三个原因：

  1. GPU 内存有限——Android 设备的 GPU 和 CPU 共享系统内存，但 GPU 可用的部分受驱动限制。Adreno 650 通常在 1GB 左右，但系统 UI、SurfaceFlinger 也在用。
  2. OCR 模型也在跑——我们的 TFLite OCR 模型如果也用 GPU Delegate，会和 llama 抢 GPU 资源。
  3. 发热控制——全 GPU 推理时设备升温最快，下一篇会详细说。

遇到的问题

问题 1：量化后的模型输出重复循环

原因：Q3 或更低的量化破坏了模型的 logits 分布，导致贪心采样总是选到同一个 token。

排查：

  1. 换用 Q4_K_M 或以上的量化级别
  2. 添加 repetition penalty（重复惩罚）：

问题 2：GGUF 模型在手机上加载极慢

加载 900MB 的 GGUF 文件时，llama_model_load_from_file 可能耗时 15-30 秒。

原因：GGUF 的 mmap 加载模式在 Android 的某些文件系统上不生效（如 scoped storage），退化为全部读入内存。

缓解措施：

  • 把 GGUF 文件放在 context.filesDir（内部存储）而不是外部存储
  • 启动时异步加载，用 Splash Screen 遮盖等待时间
  • 考虑用更小的模型文件（Q3_K_S 只有 650MB，加载快 40%）

问题 3：GPU 和 OCR 冲突导致推理结果错乱

如果 OCR 和 LLM 同时用 GPU Delegate，偶尔会出现推理结果全是乱码。

小结

Q4_K_M 量化是移动端 LLM 推理的”甜蜜点”——900MB 的体积刚好装得下，11 tok/s 的速度刚好够用，4.7/5 的质量损失几乎无感。调优的关键在于三个参数的正确组合：n_gpu_layers=16 分一半给 GPU、n_ctx=512 刚好够字幕翻译、batch=1 因为字幕是流式到达的。
